# План оптимизации межсистемного параллелизма

## Контекст

После диагностики (см. [`plans/diagnose_parallelism_v3.md`](plans/diagnose_parallelism_v3.md)) выяснили:

1. **SubWorld не нужен** — L3 cache contention не является bottleneck'ом
2. **`par_for_each_component` даёт 5x ускорение SEQ** — нужно использовать всегда
3. **Межсистемный параллелизм упирается в ~4x на 12 ядрах** из-за Rayon overhead
4. **Потолок ~4x** — фундаментальное ограничение Rayon для данного паттерна

## Новые данные: сравнение 100K vs 1000K

### 1. Overhead Rayon фиксированный, не зависит от размера данных

| Тест | 100K PAR | 1000K PAR |
|------|----------|-----------|
| 12 solo-систем | 1.53ms | 13.3ms |
| SEQ (1 поток) | 6.02ms | 55.5ms |
| Идеальный PAR (SEQ/12) | 0.50ms | 4.63ms |
| **Overhead Rayon** | **1.03ms (67%)** | **8.67ms (65%)** |

Overhead **растёт линейно** с размером данных, потому что Rayon сплиттит каждый чанк.

### 2. Расчёт количества чанков

PAR_CHUNK_SIZE = 4096:
- 100K: 12 систем × (100K / 4096) ≈ 12 × 25 = **300 чанков**
- 1000K: 12 систем × (1000K / 4096) ≈ 12 × 245 = **2940 чанков**

Каждый чанк — это задача для Rayon. 2940 задач на 12 потоков — огромный overhead на сплиттинг, распределение и синхронизацию.

### 3. Ключевой вывод

**Увеличение PAR_CHUNK_SIZE — самый простой и эффективный способ уменьшить overhead.**

Если увеличить CHUNK_SIZE в 4-16 раз:
- 100K: 300 чанков → **75 чанков** (CHUNK=16384) или **19 чанков** (CHUNK=65536)
- 1000K: 2940 чанков → **735 чанков** (CHUNK=16384) или **184 чанка** (CHUNK=65536)

## Цель

Достичь speedup 6x–8x на 12 ядрах для CPU-bound нагрузки (сейчас 4.19x).

---

## Шаг 1: Увеличить PAR_CHUNK_SIZE — первичный эксперимент

**Статус: ВЫПОЛНЕНО**

**Результаты эксперимента (100K entities, intra-system CPU-bound)**:

| CHUNK_SIZE | Speedup | Δ от 4096 |
|------------|---------|-----------|
| 4096 (baseline) | 1.49x | — |
| 16384 | **2.06x** | **+38%** |
| 32768 | 1.51x | +1% |
| 65536 | 1.17x | -21% |
| 131072 | 1.42x | -5% |

**Оптимальное значение для 100K: 16384** (2.06x, +38%).

**Изменения**:
- [`crates/apex-core/src/world.rs`](crates/apex-core/src/world.rs): `PAR_CHUNK_SIZE` изменён с `const` на `AtomicUsize`
- Добавлены: `set_par_chunk_size()`, `init_par_chunk_size_from_env()`
- `APEX_PAR_CHUNK_SIZE` env var (с `.trim()` для cmd)
- [`crates/apex-examples/examples/perf.rs`](crates/apex-examples/examples/perf.rs): отображение текущего CHUNK_SIZE

## Шаг 2: Адаптивный PAR_CHUNK_SIZE

**Статус: ВЫПОЛНЕНО**

**Формула**: `entity_count / num_threads`, clamp [4096, 65536]

**Результат**: 1.41x (сравнимо с baseline 1.49x, но хуже чем 2.06x при CHUNK=16384)

**Вывод**: Адаптивный режим не даёт преимущества для 100K. Оптимально — фиксированный CHUNK=16384.

**Изменения**:
- [`crates/apex-core/src/world.rs`](crates/apex-core/src/world.rs): `adaptive_chunk_size(entity_count, num_threads)`
- [`crates/apex-core/src/sub_world.rs`](crates/apex-core/src/sub_world.rs): все 4 parallel-метода используют adaptive_chunk_size

---

## Шаг 3: Оптимизировать `run_hybrid_parallel` — уменьшить overhead на SubWorld

**Статус: ВЫПОЛНЕНО (анализ)**

**Результат анализа**: Кеширование SubWorld не даст значимого прироста, потому что:
1. `SubWorld::new` — это просто копирование пары указателей (archetypes, archetype_indices)
2. `prepare_sub_worlds` уже кеширует `archetype_indices` через `cached_archetype_count`
3. `compute_archetype_indices` уже имеет кеш по количеству архетипов

**Решение**: Шаг пропущен — не даст измеримого прироста.

---

## Шаг 4: Эксперимент — ручное распределение систем по потокам

**Статус: ВЫПОЛНЕНО**

**Изменение**: Замена `ptrs.par_iter().enumerate().for_each(...)` на `rayon::scope(|s| { for ... { s.spawn(...) } })` в [`crates/apex-scheduler/src/lib.rs`](crates/apex-scheduler/src/lib.rs:938).

**Результаты бенчмарка (100K entities)**:

| Тест | До (par_iter) | После (scope+spawn) | Изменение |
|------|---------------|---------------------|-----------|
| 2 CPU-bound, изол. | 1.47x | 1.47x | 0% |
| 3 CPU-bound, изол. | 1.14x | 1.14x | 0% |
| 2 solo | 1.54x | 1.54x | 0% |
| 4 solo | 2.55x | 2.55x | 0% |
| 8 solo | 3.86x | 3.86x | 0% |
| 12 solo | 3.78x | 3.78x | 0% |

**Вывод**: Результаты идентичны. Оверхед `par_iter` на 12 элементах — микросекунды, время выполнения систем — миллисекунды. Основной bottleneck — sequential `for_each_component` внутри систем, а не `par_iter`.

**Изменения**:
- [`crates/apex-scheduler/src/lib.rs`](crates/apex-scheduler/src/lib.rs:938): `run_hybrid_parallel` использует `rayon::scope` + `spawn`

## Шаг 5: Бенчмарк и сравнение

**Статус: В ПРОЦЕССЕ**

### Сводная таблица результатов (100K entities)

| Оптимизация | Inter-system speedup (12 solo) | Intra-system speedup (CPU-bound) |
|-------------|-------------------------------|----------------------------------|
| Baseline (CHUNK=4096, par_iter) | 3.78x | 1.49x |
| CHUNK=16384 | 3.78x | **2.06x** |
| CHUNK=32768 | 3.78x | 1.51x |
| CHUNK=65536 | 3.78x | 1.17x |
| CHUNK=131072 | 3.78x | 1.42x |
| Adaptive CHUNK | 3.78x | 1.41x |
| scope+spawn (CHUNK=4096) | 3.78x | 1.49x |
| **scope+spawn + CHUNK=16384** | **3.78x** | **2.06x** |

### Ключевые выводы

1. **Межсистемный параллелизм (inter-system)** не изменился — упирается в ~4x на 12 ядрах
2. **Внутрисистемный параллелизм (intra-system)** улучшен с 1.49x до **2.06x** (+38%) через увеличение PAR_CHUNK_SIZE до 16384
3. **scope+spawn** не дал измеримого прироста для inter-system (оверхед par_iter на 12 задачах незначителен)
4. **Адаптивный CHUNK** не дал преимущества — оптимальный фиксированный CHUNK=16384 для 100K

### Рекомендация

Использовать комбинацию:
- `rayon::scope` + `spawn` для inter-system (код чище, предсказуемее)
- `PAR_CHUNK_SIZE=16384` для intra-system (фиксированное значение, не адаптивное)
- `APEX_PAR_CHUNK_SIZE` env var для возможности тюнинга под конкретную нагрузку
